案例研究：
歐洲領先的線上時尚平臺透過雲原生獲得突破性進展

公司  Zalando     位置  柏林, 德國     行業  線上時尚平臺

挑戰

Zalando 是歐洲領先的線上時尚平臺，自 2008 年成立以來，經歷了指數級增長。2015 年，Zalando 計劃進一步擴充套件其原有的電子商務站點，以擴充套件新的服務和產品，從而開始了徹底的變革，因此形成了自主的自組織團隊。這次擴充套件需要可以隨工程組織的增長而擴充套件的基礎架構。Zalando 的技術部門開始重寫其應用程式，使其能夠在雲端執行，並開始將其基礎架構從內部資料中心遷移到雲。雖然編排沒有立即被考慮，因為團隊遷移到亞馬遜網路服務（AWS）：“我們體驗過團隊在 AWS 上進行基礎設施架構和使用雲資源時遇到的痛苦，”開發人員生產力主管 Henning Jacobs 說，“我們遇到了一些難題。對於團隊和合規性來說，運營開銷仍然過多。為了提供更好的支援，專案引入了叢集管理。”

解決方案

該公司現在使用 Kubernetes 編排在 AWS 上執行的 Docker 容器。

影響

Jacobs 說，由於舊基礎設施“很難正確採用新技術，而 DevOps 團隊被視為瓶頸。”“現在，有了這個雲基礎架構，它們有了這種打包格式，可以包含任何在Linux核心上執行的東西。這使得很多人相當高興。工程師喜歡自主。”
“我們設想所有 Zalando 交付團隊在 Kubernetes 提供的最先進的、可靠且可擴充套件的群集基礎架構上執行其容器化應用程式。”

- Henning Jacobs, Zalando 開發人員生產力主管

當 Henning Jacobs 於2010年來到 Zalando 時，該公司剛成立兩年，有180名員工經營一家網上商店，供歐洲消費者購買時尚商品。

Zalando 的開發人員生產力主管 Jacobs 說：“它最初是一個 PHP 電子商務網站，很容易上手，但無法隨業務需求擴充套件。”

當時，公司開始從德國以外擴充套件到其他歐洲市場。快進到今天，Zalando 現在擁有超過 14000 名員工，2016 年收入為 36 億歐元，業務遍及 15 個國家/地區。他表示：“這種全面快速的增長和不斷擴充套件，一生只能有一次。”

能擁有像 Jacobs 這樣的基礎設施專家的機會是非常獨特的。就在他加入公司後，公司開始在內部重寫他們的所有應用。“這通常是我們的策略，”他表示。“例如，我們從我們自己的物流倉庫開始，但起初不知道如何做物流軟體，所以得用一些供應商軟體。然後我們用自己的軟體替換了它，因為使用現成的軟體是沒有競爭力的。需要根據特定業務需求最佳化這些流程。”

在重寫其應用程式的同時，Zalando 設定了一個目標，即從基本電子商務擴充套件到提供多租戶的平臺、大量增加的分類和樣式、當天送達，甚至您自己的平臺個人線上造型師。

擴充套件的需求最終引領了公司踏上雲原生之旅。它採用基於微服務的軟體架構，使工程團隊擁有更多的專案自主權和所有權。“遷移到雲是必要的，因為在資料中心中，團隊不可能隨心所欲。使用相同的基礎架構，因此只能執行 Java 或 Python 應用，”Jacobs 說。
“遷移到雲是必要的，因為在資料中心中，團隊不可能隨心所欲。使用相同的基礎架構，因此只能執行 Java 或 Python 應用。”
Zalando 開始將其基礎架構從兩個內部資料中心遷移到雲，這需要遷移較舊的應用程式使其在雲中準備就緒。“我們決定果斷一些，” Jacobs 說。“我們的亞馬遜網路服務基礎設施是這樣的：每個團隊都有自己的 AWS 賬戶，該賬戶是完全隔離的，這意味著沒有'提升和轉移'。基本上必須重寫應用程式，使其在雲中準備就緒，甚至到持久層也是如此。我們勇敢地回到繪圖板，重做一切，首先選擇 Docker 作為通用容器化，然後從那裡構建基礎結構。”

公司最初決定推遲編排，但隨著團隊遷移到 AWS，“我們看到團隊在 AWS 上的基礎設施和使用雲資源方面遇到了難題，” Jacobs 說。

Zalando 200多人的自主工程團隊研究使用哪些技術，並可以使用自己的 AWS 賬戶操作自己的應用程式。事實證明，此設定是一項合規性挑戰。即使制定了嚴格的遊戲規則和自動化的合規性檢查，工程團隊和 IT 合規性在解決合規性問題方面也負擔過重。Jacobs 說："違規行為會出現，我們在掃描雲基礎架構時會檢測到這些行為。“一切皆有可能，沒有強制實施，因此您必須忍受違規行為（並解決它們），而不是一開始就防止錯誤。這些都會增加團隊的開銷，以及合規性和操作的開銷。在 AWS 上啟動新的 EC2 例項也需要時間，這會影響我們的部署速度。”

Jacobs 說，團隊意識到他們需要“利用從叢集管理中獲得的價值”。當他們在2015年首次將平臺視為服務（PaaS）選項時，市場是支離破碎的；但“現在似乎有一個明確的贏家。使用 Kubernetes 似乎是一個很好的嘗試。”

Kubernetes 的過渡始於 2016 年 Zalando 的極客周期間，參與者將他們的專案部署到 Kubernetes 叢集。60名技術基礎設施部門的成員開始使用這項技術，之後每次會加入一支工程團隊。Jacobs 說：“我們總是從與他們交談開始，確保每個人的期望都清晰。然後，我們進行一些 Kubernetes 培訓，這主要是針對我們的 CI/CD 設定的培訓，因為我們的使用者介面主要透過 CI/CD 系統。但是他們必須知道Kubernetes的基本概念和API。之後每週與每個團隊同步，以檢查其進度。一旦出現什麼狀況，就可以確定我們所做的改進是否正常。”
一旦Zalando開始將應用遷移到Kubernetes，效果立竿見影。“Kubernetes 是我們無縫端到端開發人員體驗的基石。我們能夠使用單一一致且宣告性的 API 將創意運送到生產中，” Jacobs 說。
目前，Zalando正在執行最初的40個Kubernetes叢集，並計劃在可預見的將來進行擴充套件。一旦Zalando開始將申請遷移到Kubernetes，效果立竿見影。“ Kubernetes 是我們無縫端到端開發人員體驗的基石。我們能夠使用單一一致且宣告性的 API 將創意運送到生產中，” Jacobs 說。“自愈基礎架構提供了無摩擦體驗，基於低階最佳實踐構建了更高級別的抽象。我們設想所有 Zalando 交付團隊都將在 Kubernetes 提供的最先進的可靠和可擴充套件群集基礎架構上執行其容器化應用程式。” Jacobs 說，使用舊的內部基礎設施，“很難正確採用新技術，而 DevOps 團隊被視為瓶頸。”“現在，有了這個雲基礎架構，它們有了這種打包格式，可以包含執行在 Linux 核心中的任何內容。這使得很多人相當高興。工程師喜歡自主性。”在Zalando的Kubernetes實施中出現了一些挑戰。Jacobs 說：“我們是一支由七人組成的團隊，為不同的工程團隊提供叢集，我們的目標是為所有團隊提供堅如磐石的體驗。”“我們不想要寵物叢集。我們不想了解他們的工作量；它應該只是開箱即用。考慮到這一點，叢集自動縮放非常重要。執行叢集管理的方法有很多種，這不是核心的一部分。因此，我們建立了兩個元件來預配群集，具有叢集的登錄檔，並管理整個叢集生命週期。”

Jacobs 的團隊還致力於改進Kubernetes-AWS 整合。“由於許多限制條件,需要透過基礎設施來擴充套件每個自主團隊的想法。” 此外，“仍然缺少很多最佳實踐，”Jacobs說。例如，該團隊最近解決了 pod 安全策略問題。“在Kubernetes已經有一個概念，但沒有記錄，所以有點棘手，”他說。大型的Kubernetes社群是解決這個問題的一大幫助。為了幫助其他公司走上同樣的道路，Jacobs在一份名為“在生產中執行 Kubernetes ”的檔案中彙編了他的團隊的經驗教訓。
“ Kubernetes API 允許我們以與雲提供商無關的方式執行應用程式，這使我們能夠在未來幾年中自由訪問 IaaS 提供商...我們期望 Kubernetes API 成為 PaaS 基礎設施的全球標準，並對未來的繼續旅程感到興奮。”
最後，Kubernetes 使 Zalando 能夠引進和維護公司為發展其平臺而設想的新產品。Jacobs 說：“時尚諮詢產品使用 Scala，而我們以前的基礎設施也難以實現這一點。”這是一個解決方法，該團隊需要平臺團隊提供越來越多的支援，只是因為他們使用了不同的技術。現在有了Kubernetes，它就自主了。無論工作負載是什麼，該團隊都可以走自己的路，而 Kubernetes 可以防止其他瓶頸。

展望未來，Jacobs 將 Zalando 的新基礎設施視為公司在進行的其他工程中的巨大推動因素，從新的物流軟體到連線品牌的平臺功能，以及資料科學家夢寐以求的產品。Jacobs說：“一個願景是，如果你看下一部 James Bond 的電影，看看他穿的西裝，你就應該能夠自動訂購，並在一小時內把它送到你身邊。”“這是關於連線整個時尚領域。如果您遇到瓶頸，因為每個人都在同一個資料中心執行，因此限制很大，則這絕對是不可能的。需要基礎設施來擴充套件每個自主團隊的想法。”

對於考慮這項技術的其他公司，Jacobs 說，他不一定建議像 Zalando 那樣做。他表示：“如果你準備嘗試失敗，那麼這樣做是可以的。”“設定正確的期望是必須的。並不是一切都會起作用。重寫應用和這種型別的組織更改可能會造成中斷。我們移動的第一個產品至關重要。存在大量依賴關係，而且時間比預期長。也許我們應該從不那麼複雜、不是業務關鍵的東西開始，只是為了開個好頭。”

但是，一旦他們到了另一邊，“每個人都很清楚，沒有大的選擇，” Jacobs 補充說。“ Kubernetes API 允許我們以與雲提供商無關的方式執行應用程式，這使我們能夠在未來幾年中自由訪問 IaaS 提供商。Zalando 受益於遷移到 Kubernetes，因為我們能夠利用現有知識建立工程平臺，為我們的工程師提供靈活性和速度，同時顯著降低運營開銷。我們期望 Kubernetes API 成為 PaaS 基礎設施的全球標準，並對未來的旅程感到興奮。”
