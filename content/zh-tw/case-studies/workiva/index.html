案例研究：
使用 OpenTracing 幫助查詢瓶頸

公司  Workiva     Location  艾姆斯，愛荷華州     行業  企業軟體

挑戰

Workiva提供了一個基於雲的平臺，用於管理和報告業務資料。此 SaaS 產品 Wdesk 被 70% 以上的財富 500 強企業使用。隨著公司從單體系統向更分散的基於微服務的系統轉變，“我們有許多人在研究這個問題，他們都在不同的團隊中，所以我們需要確定問題是什麼，瓶頸在哪裡，”高階軟體架構師 MacLeod Broad 說。隨著後端程式碼在 Google App Engine，Google Compute Engine，以及 Amazon Web Services 上執行，Workiva 需要一個還未被開發的跟蹤系統。在準備公司首批使用 AWS 的產品時，Broad 的團隊將新應用程式中構建的電子表格資料與 Workiva 現有系統上的舊應用程式中建立的文件相關聯，該功能涉及“同步和連結”功能，從而發現了一個理想的跟蹤用例：存在迴圈依賴關係，而最佳化通常證明是不影響整體速度的微最佳化。

解決方案

Broad 的團隊推出了與平臺無關的分散式跟蹤系統 OpenTracing，幫助他們找出瓶頸。

影響

現在在整個公司使用，OpenTracing 產生了立竿見影的效果。軟體工程師 Michael Davis 報告說：“跟蹤讓我們能夠立即瞭解如何改進我們的服務。透過檢視每個呼叫的花費時間以及最常使用哪些呼叫的組合，我們能夠在單個修復中將平均響應時間減少 95%（從 600ms 到 30ms）。”
使用 OpenTracing，我的團隊能夠檢視跟蹤，並針對其他團隊提出最佳化建議，而無需檢視他們的程式碼。
— MacLeod Broad, Workiva 高階軟體架構師

去年秋天，MacLeod Broad 在 Workiva 的平臺團隊在準備該公司的首批產品之一，當遇到障礙時就使用 Amazon Web Services 。

早期，Workiva 的後端主要執行在Google App Engine上。但隨著 Workiva 的 SaaS 產品，Wdesk，一種基於雲的管理和報告業務資料平臺，將客戶群增長到財富 500 強公司的 70% 以上，情況發生了變化。“隨著客戶需求的增長和產品供應的擴大，我們開始利用更廣泛的服務，如 Amazon Web Services 以及其他 Google Cloud Platform 服務，從而建立一個多供應商環境。” 有了這個新產品，具備“同步和連結”功能，透過它，資料“經歷了一大堆服務，從新的電子表格系統[Amazon Aurora]到我們所謂的連結系統，然後透過 http 推送到我們現有的系統，然後進行一些計算，結果將傳回新系統，”Broad 說。“我們當時進行最佳化是為了速度。我們認為做了這種極好的最佳化，然後它會變成一個微觀最佳化，這並沒有真正影響系統的整體速度。”

Broad 團隊面臨的挑戰聽起來可能為其他公司所熟悉，這些公司也從單體系統轉向更分散的基於微服務的系統。Broad 說：“我們有許多人從事這方面的工作，他們都在不同的團隊中，因此很難了解問題是什麼以及瓶頸在哪裡。”

“每個服務團隊都經歷著不同的架構迭代，很難了解每個團隊系統中的實際內容，”他補充道。“我們有迴圈依賴關係，其中有三個或四個不同的服務團隊不確定問題的真正位置，需要大量的來回溝通。因此，我們浪費了很多時間說，‘這個哪一部分比較慢？根據應用場景的不同，哪一部分有時很慢？隨著時間推移，哪一部分會逐漸變慢？這個程序的哪一部分是非同步的，因此，它是否長時間執行並不重要？我們在做的哪些是多餘的，其中哪一部分是錯誤？’”
“跟蹤系統可以一目瞭然地解釋體系結構，縮小效能瓶頸並歸零，通常只是幫助指導高級別的調查。一眼就能做到這一點，比在會議或三天的除錯中要快得多，而且比從來不找出問題得過且過要快得多。”
— MACLEOD BROAD, WORKIVA 高階軟體架構師
簡而言之，它是跟蹤的理想用例。Broad 說：“跟蹤系統可以一目瞭然地解釋體系結構，縮小效能瓶頸並歸零，通常只是幫助在高級別指導調查。”“一眼就能做到這一點，比在會議或三天的除錯中要快得多，而且比從不找出問題得過且過要快得多。”

Workiva 的後端程式碼在 Google Compute Engine 、App Engine 和 AWS 上執行，Broad 知道他需要一個與平臺無關的跟蹤系統。“我們正在研究不同的追蹤解決方案，”他說，“我們決定，由於市場似乎是一個不斷髮展的市場，我們不想被一個供應商卡住。因此，OpenTracing 似乎是避免供應商限制我們實際使用的後端的最簡捷方法。”

Broad 說，一旦他們將 OpenTrace 引入到第一個用例中，“跟蹤使得瓶頸所在位置變得非常明顯。”儘管每個人都認為是 Workiva 的現有程式碼減緩了速度，但事實並非如此。Broad 說：“看起來現有程式碼速度很慢，只是因為它能夠達到到我們下一代服務的水平，而且它們需要很長時間才能處理所有這些請求。”“在瀑布圖上，你可以看到每個請求在得到響應時所做的完全相同的工作。因此，對於被分頁的每個響應，每個服務請求看起來都完全相同。然後，就不用費神去思考‘為什麼它再次做所有這些工作？’”

使用 OpenTrace 的跟蹤功能，“我的團隊能夠檢視跟蹤，並針對另一個團隊提出最佳化建議，而無需檢視他們的程式碼，”Broad 說。“我們命名跟蹤的方式可以說明請求是執行 SQL 呼叫還是建立 RPC。因此，可以非常簡單地說，‘好吧，我們知道它能處理所有這些請求。處理一次快取一次。’我們基本上完成了。所有這些呼叫立即變成了亞秒級的呼叫。”

“我們正在研究不同的跟蹤解決方案，我們決定使用 OpenTracing，由於市場似乎是一個不斷髮展的市場，我們不想被一個供應商卡住。因此，OpenTracing 似乎是避免供應商限制我們實際使用的後端的最簡捷方法。”
— MACLEOD BROAD, WORKIVA 高階軟體架構師
在第一個用例成功後，參與嘗試的每個人都回去重新編排了他們的產品。跟蹤功能已新增到幾個更多的用例中。Broad 說：“我們希望儘早度過最初的實施難關，而不會讓整個部門一起渡過難關。”“現在，許多團隊在啟動新服務時新增它。我們現在確實比以前更將推動採用。”

一些團隊很快就贏了。軟體工程師 Michael Davis 說：“跟蹤讓我們能夠立即瞭解如何改進我們的（工作空間）服務。透過檢視每個呼叫的花費時間以及最常使用哪些呼叫的組合，我們能夠在單個修復中將平均響應時間減少 95%（從 600ms 到 30ms）”。

Workiva 的大部分主要產品現在都使用 OpenTracing 進行跟蹤，將資料推送到 Google StackDriver。即使未完全使用跟蹤功能的產品，也具有一些元件和庫。

Broad 指出，由於一些工程師正在使用 App Engine，並且已經擁有平臺的 Appstats 庫分析效能的經驗，因此他們不需要花太多時間才能習慣使用 OpenTracing。但其他人則有點不情願。“我認為，使用 OpenTracing 的最大障礙是擔心引入跟蹤和 StackDriver 的成本會是多少，”他說。“人們也非常關心將中介軟體新增到他們正在處理的任何內容中。有關傳遞上下文以及如何完成這些工作的問題很常見。我們的許多 Go 開發人員對此都很好，因為他們已經以這樣或那樣的形式這樣做了。我們的 Java 開發人員並不熱衷於這樣做，因為他們使用了其他不需要該功能的系統。”

但好處顯然超過了人們的擔憂，而今天，Workiva 的官方政策是使用追蹤。 事實上，Broad 認為跟蹤天然符合 Workiva 現有的日誌記錄和指標系統。“這是我們在內部展示的方式，也是我們設計使用方式的方式，”他說。“我們的跟蹤記錄在與我們的應用指標和日誌記錄資料完全相同的機制中，並且它們被推送的方式完全相同。因此，在建立和記錄所有資料時，我們對待這些資料完全一樣。我們有一個內部庫，用於日誌記錄、遙測、分析和跟蹤。”
“跟蹤讓我們能夠立即、可操作地洞察如何改進我們的（工作空間）服務。透過檢視每個呼叫花費的時間以及最常使用哪些呼叫的組合，我們能夠在單個修復中將平均響應時間減少 95%（從 600ms 到 30ms）。”
— Michael Davis, 軟體工程師, Workiva
對於 Workiva，OpenTracing 已成為一種重要工具，透過觀察使用模式來聚焦最佳最佳化和確定什麼是微觀最佳化。Broad 說：“在某些專案中，我們經常假設客戶正在做什麼，我們針對這些瘋狂的測試案例進行最佳化，這些案例的 1% 是同用戶實際非常相符的。”“這樣是非常有益的，‘好吧，我們在每個執行 X 的請求上新增 100 毫秒，如果最壞的情況，我們也只需要新增 100 毫秒，這僅發生在千分之一的請求或一百萬個請求中的一個。’”

與許多其他公司不同，Workiva 還跟蹤客戶端。Broad 說：“對我們來說，使用者體驗很重要，如果 RPC 執行後還需要 5 秒才能在瀏覽器中顯示，則 RPC 執行需要 100 毫秒就不重要了。”“因此，對於我們而言，這些客戶端響應時間非常重要。我們跟蹤它，看看載入的哪些部分需要很長時間。我們正在研究什麼是“載入”的定義。是當你訪問到它，還是當它被渲染時，或者當你可以與它互動時？我們計劃使用跟蹤來關注和更好地瞭解這些內容。”

這還需要根據外部和內部時鐘的差異進行調整。“在時間糾正之前，這是非常恐怖的；我們的跟蹤比什麼都更具有誤導性，” Broad 說。“因此，我們決定在響應標頭上返回一個時間戳，然後讓客戶端根據該時間調整其時間，而不是更改其內部時鐘，而只需計算客戶端收到時響應時間的偏移量。如果最終出現一種不可能的情況，客戶端的 RPC 呼叫持續了 210 毫秒但響應返回時間並不在這個範圍內，那就必須得重新呼叫請求。”

Broad 對 OpenTracing 對公司產生的影響感到興奮，並且也在展望該技術還能實現什麼。一種可能性是使用跟蹤實時更新文件。他表示：“使文件與現實保持同步是一項重大挑戰。”“例如，我們剛剛運行了跟蹤模擬，或者我們剛剛在此新部署上運行了煙霧測試，但是結果顯示架構與文件不匹配。我們可以找到是誰的責任，並通知他們進行更新。這是我希望在未來透過追蹤獲得的功能之一。”
