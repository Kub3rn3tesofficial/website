案例研究:
使小型分散式團隊輕鬆部署

公司 Buffer     位置  全球     行業  社交媒體技術公司

挑戰

Buffer 擁有一支80人的分散式團隊，他們遍佈全球近十幾個時區。這樣一個為代理商和營銷人員提供社交媒體管理的公司，希望解決其“典型的單一龐大編碼基數問題”，架構師 Dan Farrelly 這樣說。“我們希望擁有一種流動性的基礎架構，開發人員可以建立一個應用程式，可以根據需要部署並橫向擴充套件它”。

解決方案

擁抱容器化，Buffer 將其基礎設施從 AWS 上的 Elastic Beanstalk 遷移到由 Kubernetes 負責編排的 Docker 上。

影響

Farrelly 說，新系統“提高了我們將新變化進行部署和上線的能力”。“在自己的計算機上構建一些東西，並且知道它是可用的，這已經讓事情簡單了很多；而且我們的反饋週期現在也快了很多。”
“我們的團隊能夠直接使用既有的 Kubernetes 解決方案，這太棒了，而且它還在不斷改進中。在意識到我們需要某些功能之前，它就出現在下一個版本中，或者在未來幾個月內出現。

- DAN FARRELLY, BUFFER ARCHITECT

Dan Farrelly 用木工類比來解釋他的公司 Buffer ，隨著過去幾年的發展，公司開始有這個問題。

“如果你自己做一張桌子，這很好！”公司的架構師說。“如果你請另一個人一起來做這個桌子，也許這個人可以在你拋光桌面時開始對腿進行拋光。但是，當你把第三個或第四個人帶進來時，有人也許應該在另外一張桌子上工作。”需要處理越來越多不同的桌子，使 Buffer 走上了 Kubernetes 實現微服務和容器化的道路。

自2012年左右以來，Buffer 開始使用 Elastic Beanstalk ,這是亞馬遜網路服務提供的網路基礎設施編排服務。“我們部署了一個單一的PHP應用程式,它是在五六個環境中相同的應用程式，”Farrelly 說。“我們在很大程度上是一家產品驅動型的公司。” 這一切都是為了儘快給應用推出新功能並進行交付，如果能夠正常執行，我們就不會花太多時間在上面。如果產品變得有點慢，我們可能會使用更快的伺服器或只是增加一個例項，這些就已經足夠了。然後繼續前進。

但事情在2016年就到了頭。隨著應用提交修改的數量不斷增加，Farrelly 和 Buffer 當時的首席技術官 Sunil Sadasivan 決定,是重新思考和構建其基礎架構的時候了。“這是一個典型的單一龐大編碼基數問題，”Farrelly說。公司的一些團隊已經在開發環境中成功使用Docker，但在生產環境中，執行在 Docker 上的唯一應用程式是一個看不到真實使用者流量的營銷網站。他們希望在使用 Docker 上更進一步,下一步是尋找業務流程的選項。
Kubernetes 所做的所有事情都很好地適應了 Buffer 的需求。Farrelly 說：“我們希望擁有一種流動基礎架構，開發人員可以建立一個應用程式，並在必要時部署並進行水平擴充套件。”“我們很快就使用一些指令碼來設定幾個測試叢集，在容器中構建了一些小的概念性驗證應用程式，並在一小時內部署了這些內容。我們在生產中執行容器方面經驗很少。令人驚奇的是，我們能很快地用 Kubernetes 來處理。”
首先，他們考慮了Mesosphere、DC/OS和Amazon Elastic Container Service(他們的資料系統團隊已經將其用於某些資料管道作業)。雖然他們對這些產品印象深刻，但他們最終還是與 Kubernetes 合作。 Farrelly 說：“我們的應用仍在 AWS 上執行，但是我們的團隊透過 Kubernetes 無需手動配置即可建立服務並按需建立負載均衡器，這是使用 Kubernetes 的最佳入門體驗。”“我們不需要考慮如何配置這個或那個,特別是相較於以前的 Elastic Beanstalk 環境，它為我們提供了一個自動配置的負載均衡器。我真的很喜歡 Kubernetes 對命令列的控制，只需要對埠進行配置，這樣更加靈活。Kubernetes 是為做它所做的事情而設計的,所以它做得非常好。”

Kubernetes 所做的所有事情都很好地適應了 Buffer 的需求。Farrelly 說：“我們希望擁有一種流動基礎架構，開發人員可以建立一個應用程式，並在必要時部署並進行水平擴充套件。”“我們很快就使用一些指令碼來設定幾個測試叢集，在容器中構建了一些小的概念性驗證應用程式，並在一小時內部署了這些內容。我們在生產中執行容器方面經驗很少。令人驚奇的是,我們能很快地用 Kubernetes 來處理。”

最重要的是，它為公司最顯著的特徵之一提供了強大的解決方案：他們的遠端團隊分佈在十幾個不同的時區。Farrelly 說：“對我們的基礎設施有深入瞭解的人生活在不同於我們相對集中的時區，而我們的大部分產品工程師都住在其他地方。”“因此，我們確實希望有人能夠儘早掌握這套系統並利用好它，而不必擔心部署工程師正在睡覺。否則，人們會為了一些東西必須得等待12到24小時左右。看到人們前進得更快，這真的很酷。”

Farrelly 說，Buffer 擁有相對較小的工程團隊，只有 25 人，只有少數人從事基礎設施工作，大多數前端開發人員都需要“一些強大的配置能力，以便部署任何他們想要的內容。”以前，“只有幾個人知道如何用舊的方式設定一切。有了這個系統，審查文件變得很容易，並且能很快地看到效果。它降低了我們獲取在開發環境中所需要一切的門檻。因為我們團隊的人數不足以來構建所有這些工具或像其他大公司那樣管理基礎架構。”
Farrelly 說：“在我們以往的工作方式中，反饋迴圈流程要長得多，而且很微妙，因為如果你部署了某些東西，就存在破壞其他東西的風險。”“我們透過圍繞 Kubernetes 構建的部署型別，能夠及時檢測 Bug 並修復它們，並使其部署得超快。這一秒正在修復漏洞，不一會就上線運行了。”
為了幫助解決這一問題，Buffer 開發人員編寫了一個部署機器人，該機器人包裝了 Kubernetes 部署過程，並且每個團隊都可以使用。”“以前,我們的資料分析師會更新 Python 分析指令碼，並且必須等待該團隊的主管單擊該按鈕並部署它，”Farrelly 解釋道。“現在，我們的資料分析師可以進行更改，輸入 Slack 命令，‘/deploy’,它會立即進行部署。他們不需要等待這些緩慢的週轉時間，他們甚至不知道它在哪裡執行。這些都不重要了。

團隊使用 Kubernetes 從頭開始構建的第一個應用程式是一種新的影象大小調整服務。作為一種社交媒體管理工具，它允許營銷團隊透過發帖進行協作，並透過多個社交媒體配置檔案和網路傳送更新，Buffer 必須能夠根據需要調整照片的大小，以滿足不同的社交網路。Farrelly 說：“我們一直有這些拼湊在一起的解決方案。”

為了建立這項新服務，一位高階產品工程師被指派學習 Docker 和 Kubernetes，然後構建服務、測試、部署和監視服務，他能夠相對快速地完成該服務。Farrelly說：“在我們以往的工作方式中，反饋迴圈流程要長得多，而且很微妙，因為如果你部署了某些東西，就存在破壞其他東西的風險。”“我們透過圍繞 Kubernetes 構建的部署型別，能夠及時檢測 Bug 並修復它們，並使其部署得超快。這一秒正在修復漏洞,不一會就上線運行了。”

此外，與舊系統不同，他們只需一個命令就可以水平縮放內容。“當我們推出它,”Farrelly說，“我們可以預測，只需點選一個按鈕。這使我們能夠處理使用者對系統的需求,並輕鬆擴充套件以處理它。”

他們以前不能做的另外一件事是金絲雀部署。Farrelly說，這種新功能“使我們在部署重大變革方面更加自信。”“以前，雖然進行很多測試但仍表現不錯，但它也存在很多‘手指交叉’。這是每天執行 800000 次的東西，這是我們業務的核心。如果它不工作，我們的業務也無法執行。在 Kubernetes 世界中，我可以執行金絲雀部署以測試 1%，如果它不工作，我可以很快將其關閉。這使我們在快速部署和推出新更改的同時降低風險的能力得到了提高。”
Farrelly 說：“如果你想在生產中執行容器，擁有像谷歌內部使用的那種效果，那麼 Kubernetes 就是一個很好的方法。”“我們是一個相對較小的團，實際上執行 Kubernetes，我們之前從來沒來做過這樣的事情。因此，它比你想象的更容易上手，這正是我想要告訴正在嘗試使用它的人們的一件很重要的事。挑幾個應用,把它們準備好,在機器上執行幾個月,看看它能處理的怎麼樣。透過這種方式你可以學到很多東西。”
到 2016 年 10 月，Buffer 54% 的流量都透過其 Kubernetes 叢集。Farrelly 說：“我們很多傳統功能仍然執行正常，這些部分可能會轉移到 Kubernetes 或永遠留在我們的舊設定中。”但該公司當時承諾，“所有新的開發，所有新功能,將在Kubernetes上執行。”

2017年計劃是將所有舊應用程式遷移到新的 Kubernetes 叢集，並執行他們從舊基礎架構中撤出的所有內容，以及他們正在另一個 Kubernetes 叢集上開發的新服務。Farrelly 說：“我想為團隊中的每個人帶來我們早期服務的所有好處。”

對於 Buffer 的工程師來說，這是一個令人興奮的過程。“每次部署新服務時，我們都需要弄清楚:好的，體系結構是什麼？這些服務如何溝通？構建此服務的最佳方式是什麼？”Farrelly說。“然後，我們使用 Kubernetes 的特性將所有部分聚合到一起。在學習如何設計面向服務的體系結構時，它使我們能夠進行試驗。以前，我們只是不能做到這一點。這實際上給了我們一個空白的白板，所以我們可以做任何我們想要的。”

部分空白是 Kubernetes 提供的靈活性，如果某一天 Buffer 可能想要或需要改變他的雲服務。“這是與雲無關的，所以也許有一天我們可以切換到谷歌或其他地方，”Farrelly 說。“我們非常依賴亞馬遜的基礎服務，但很高興知道，如果我們需要的話，我們可以搬走。”

此時，Buffer 團隊無法想象以任何其他方式執行其基礎結構，他們很樂意傳播這一資訊。Farrelly 說：“如果你想在生產中執行容器，擁有像谷歌內部使用的那種效果，那麼 Kubernetes 就是一個很好的方法。”“我們是一個相對較小的團隊，實際上執行 Kubernetes，我們之前從來沒來做過這樣的事情。因此，它比你想象的更容易上手，這正是我想要告訴正在嘗試使用它的人們的一件很重要的事。挑幾個應用，把它們準備好，在機器上執行幾個月，看看它能處理的怎麼樣。透過這種方式你可以學到很多東西。”
