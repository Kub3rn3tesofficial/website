Kubernetes Pods

在模組 2建立 Deployment 時, Kubernetes 添加了一個 Pod 來託管你的應用例項。Pod 是 Kubernetes 抽象出來的，表示一組一個或多個應用程式容器（如 Docker），以及這些容器的一些共享資源。這些資源包括:

  • 共享儲存，當作卷
  • 網路，作為唯一的叢集 IP 地址
  • 有關每個容器如何執行的資訊，例如容器映像版本或要使用的特定埠。

Pod 為特定於應用程式的“邏輯主機”建模，並且可以包含相對緊耦合的不同應用容器。例如，Pod 可能既包含帶有 Node.js 應用的容器，也包含另一個不同的容器，用於提供 Node.js 網路伺服器要釋出的資料。Pod 中的容器共享 IP 地址和埠，始終位於同一位置並且共同排程，並在同一工作節點上的共享上下文中執行。

Pod是 Kubernetes 平臺上的原子單元。 當我們在 Kubernetes 上建立 Deployment 時，該 Deployment 會在其中建立包含容器的 Pod （而不是直接建立容器）。每個 Pod 都與排程它的工作節點繫結，並保持在那裡直到終止（根據重啟策略）或刪除。 如果工作節點發生故障，則會在叢集中的其他可用工作節點上排程相同的 Pod。

工作節點

一個 pod 總是執行在 工作節點。工作節點是 Kubernetes 中的參與計算的機器，可以是虛擬機器或物理計算機，具體取決於叢集。每個工作節點由主節點管理。工作節點可以有多個 pod ，Kubernetes 主節點會自動處理在叢集中的工作節點上排程 pod 。 主節點的自動排程考量了每個工作節點上的可用資源。

每個 Kubernetes 工作節點至少執行:

  • Kubelet，負責 Kubernetes 主節點和工作節點之間通訊的過程; 它管理 Pod 和機器上執行的容器。
  • 容器執行時（如 Docker）負責從倉庫中提取容器映象，解壓縮容器以及執行應用程式。

如果它們緊耦合並且需要共享磁碟等資源，這些容器應在一個 Pod 中編排。

使用 kubectl 進行故障排除

在模組 2,你使用了 Kubectl 命令列介面。 你將繼續在第3單元中使用它來獲取有關已部署的應用程式及其環境的資訊。 最常見的操作可以使用以下 kubectl 命令完成：

  • kubectl get - 列出資源
  • kubectl describe - 顯示有關資源的詳細資訊
  • kubectl logs - 列印 pod 和其中容器的日誌
  • kubectl exec - 在 pod 中的容器上執行命令

你可以使用這些命令檢視應用程式的部署時間，當前狀態，執行位置以及配置。

現在我們瞭解了有關叢集元件和命令列的更多資訊，讓我們來探索一下我們的應用程式。
